# Deep-Dive：东东深度研究学习工作流

> 一句话本质：用"研究-学习-费曼-应用"四步循环，把任何主题变成可落地、可引用、可复用的知识资产。

## 触发识别

**触发关键词**：
- "dd"（推荐，最简洁）
- "deep-dive"
- "帮我deep-dive"
- "深度研究一下"
- "全面学习"
- "费曼输出"

**输入类型**：
- 主题词（如"AI原生组织"）
- 链接（文章/论文/视频URL）
- 图片（截图、图表）

## 查询澄清机制（v1.4.1 新增）

> 借鉴 OpenAI 多智能体研究框架的"澄清代理"设计：在启动深度研究前，先优化查询精度，确保研究方向正确。

### 何时执行澄清

**必须澄清**（以下任一条件满足）：
- 主题词模糊（如"AI应用"范围太广）
- 东东提供的链接/图片主题不明确
- 涉及多个可能的研究方向（如"研究特斯拉"可能是公司/科学家/单位）
- 东东说"了解一下"但无具体范围

**可选澄清**（根据上下文判断）：
- 主题有一定明确性，但边界可进一步聚焦
- 东东时间有限，需要确定研究深度

**跳过澄清**（直接进入研究）：
- 东东明确说"深度研究XX"且主题具体（如"dd transformer架构"）
- 东东提供了明确的上下文和范围
- 是之前研究的延续（如"继续研究刚才那个"）

### 澄清执行流程

```
东东输入主题/链接/图片
    ↓
[自动判断] 是否需要澄清？
    ↓ 是
生成2-3个关键澄清问题
    ↓
向东东提问（简洁友好）
    ↓
东东回答
    ↓
整合回答，生成优化后的研究指令
    ↓
[可选] 向东东确认研究方向
    ↓
进入五阶段工作流
```

### 澄清问题设计原则

**有效的澄清问题**（参考OpenAI指令构建代理）：
1. **聚焦关键维度**：研究范围、时间框架、应用场景、目标受众
2. **避免冗余**：不问已明确的信息
3. **提供选项**：复杂选择给出建议选项
4. **设定默认**：说明"如无偏好，将研究XX方面"

**示例澄清对话**：

| 东东输入 | 澄清问题 | 优化后指令 |
|----------|----------|------------|
| "研究一下AI" | "您希望关注哪个维度？技术原理/商业应用/行业影响？是否有特定行业偏好？" | "研究AI在企业管理领域的商业应用，重点关注2024-2026年的落地案例和投资趋势" |
| "看看这篇文章" | "您希望我：1) 总结文章核心观点；2) 扩展研究文章主题；3) 分析文章方法论？" | "基于文章内容，扩展研究'RAG优化技术'主题，重点关注向量数据库选型策略" |
| "了解一下马斯克" | "您是指：1) 埃隆·马斯克个人管理风格；2) 特斯拉/SpaceX商业模式；3) 其在AI领域的布局？" | "研究埃隆·马斯克的管理决策模式，重点关注第一性原理在其商业决策中的应用" |

### 研究指令优化标准

澄清完成后，生成结构化的研究指令：

```markdown
## 优化后的研究指令

**研究主题**：[具体主题]
**研究范围**：
- 包含：[明确包含的内容]
- 不包含：[明确排除的内容]
**时间框架**：[如：2024-2026年最新进展 / 历史演进 / 不限时间]
**深度要求**：[概览级 / 应用级 / 原理级]
**应用场景**：[东东将用于什么场景]
**输出格式**：[标准Deep-Dive文档 / 速查卡片 / PPT大纲]

**研究子目标**：
1. [子目标1]
2. [子目标2]
3. [子目标3]
```

## 五阶段工作流（Planner → Executor → Checker）

基于2026年AI Agent最佳实践，采用"规划-执行-校验"三段式确保研究质量：

```
┌─────────────────────────────────────────────────────────────┐
│ Phase 1: Planning（规划）- 智能拆解研究目标                  │
│ • 将主题拆解为3-5个可执行研究子目标                          │
│ • 生成结构化、可解释的研究路径                               │
│ • 规划信息整合策略与工具选择                                 │
│ • 设定每个子目标的预期产出与验收标准                         │
│ • 标记Known Unknowns（已知知识缺口）                         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 2: Research（研究）- 多源探索与证据收集                │
│ • 多源搜索（10-15个高质量来源）                              │
│ • 优先一手来源：arXiv论文、官方报告、专家原话               │
│ • 交叉验证核心观点，标记矛盾与共识                           │
│ • 多模态信息采集：文本+图表+数据+代码                        │
│ • 中间产物存档：保存关键来源、原始数据、初步发现             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 3: Learning（学习）- 结构化萃取与知识图谱构建          │
│ • 提取核心概念与定义                                         │
│ • 梳理逻辑框架与因果关系                                     │
│ • 识别底层原理与第一性原理                                   │
│ • 构建知识图谱：实体-关系-实体三元组可视化                   │
│ • 记录"明确不知道"的边界（负向知识）                         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 4: Feynman（费曼）- 多维度简化与对抗性验证             │
│ • 概念拆解：识别核心要素与依赖关系                           │
│ • 零基础阐述：用初中生能懂的语言重构                         │
│ • 多模态转译：文字→图表→公式→代码                            │
│ • 对抗性验证：模拟质疑者角色，暴露逻辑漏洞                   │
│ • 类比创造：结构映射（标注类比失效边界）                     │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 5: Verification（校验）- 质量自检与可复现性验证        │【新增】
│ • 结构完整性检查：所有必填字段是否完备                       │
│ • 内容一致性检查：前后观点是否矛盾                           │
│ • 事实准确性检查：核心数据是否有来源支撑                     │
│ • 可复现性检查：研究路径是否可追溯                           │
│ • 降级策略：若校验不通过，返回对应阶段重跑                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Phase 6: Application（应用）- 场景化落地                     │
│ • 应用场景（3-5个具体场景，含边界条件）                      │
│ • 行动建议（可执行的步骤，含验证方法）                       │
│ • 能力引用（如何引用这份知识提升什么能力）                   │
│ • 反事实思考：如果不采用这个方案会怎样？                     │
└─────────────────────────────────────────────────────────────┘
```

### 知识资产沉淀（Asset Keep）

> 基于SPARK模型的Keep阶段，将研究成果转化为可复用资产。这是从"单次学习"到"知识复利"的关键。

**资产类型与创建指南**：

| 资产类型 | 判断标准（何时创建） | 创建步骤 | 存储位置 |
|----------|---------------------|----------|----------|
| **检查清单**<br>(Checklist) | 研究主题涉及多步骤操作或易遗漏点 | 1. 列出关键步骤/检查点<br>2. 标注常见错误<br>3. 设计勾选格式 | `memory/checklists/{主题}.md` |

| **决策框架**<br>(Framework) | 研究主题涉及选择/判断场景 | 1. 明确决策维度<br>2. 设计评分标准或if-then规则<br>3. 给出应用示例 | `memory/frameworks/{主题}.md` |

### 科学技能蒸馏（v1.4.1 新增）

> 借鉴 S1-NexusAgent 框架：将高质量研究路径提炼为可复用的"科学技能"，实现从"执行-学习"到"执行-学习-复用"的进化。

**什么是技能蒸馏**：
将一次成功的深度研究中，**研究策略、验证方法、知识提取模式**提炼为可复用的模板，供类似主题直接使用。

**何时进行技能蒸馏**：
- 研究质量评分 ≥ 8分（优秀或良好）
- 研究主题具有代表性（可能重复出现）
- 研究过程中发现了有效的独特方法

**可蒸馏的技能类型**：

| 技能类型 | 蒸馏内容 | 存储位置 | 示例 |
|----------|----------|----------|------|
| **搜索策略模板** | 该主题的有效搜索路径、关键词组合、来源优先级 | `memory/skills/search/{主题类别}.md` | "研究AI公司时，优先搜索arXiv技术报告+官方博客+创始人访谈" |
| **验证检查清单** | 该主题的事实验证要点、常见误区、可信度评估标准 | `memory/skills/verify/{主题类别}.md` | "验证AI性能声明时，必查：训练数据规模、测试集选择、对比基准" |
| **知识提取框架** | 从该主题材料中提取关键信息的标准结构 | `memory/skills/extract/{主题类别}.md` | "分析商业模式时，提取：收入来源、成本结构、护城河、增长飞轮" |
| **费曼转换模式** | 将该主题专业知识转换为通俗解释的有效类比 | `memory/skills/feynman/{主题类别}.md` | "解释分布式系统时，用'邮递员送信'类比一致性模型" |

**技能蒸馏执行流程**：

```
完成高质量Deep-Dive（评分≥8分）
    ↓
分析成功要素（哪些策略特别有效）
    ↓
提炼可复用模式（抽象为通用模板）
    ↓
创建技能文件（按类型存储）
    ↓
在asset-index.md中登记
    ↓
下次类似主题时优先调用
```

## 五、待深究的问题（Known Unknowns）

> 明确记录当前研究未覆盖、需要进一步探索的问题。这不是缺陷，而是诚实标记知识边界，为后续研究留下线索。

### 问题分类标准

| 类型 | 定义 | 示例 |
|------|------|------|
| **数据缺口** | 需要但当前未找到的数据/统计 | "缺少2026年云南地区AI应用的渗透率数据" |
| **观点冲突** | 不同来源观点矛盾，未充分验证 | "论文A认为X有效，论文B认为X无效，需要进一步验证" |
| **边界模糊** | 适用边界不清晰 | "不清楚这个方法在小团队（<5人）中是否适用" |
| **因果未明** | 相关性已知，因果机制不清 | "观察到X和Y相关，但因果链条未完全理清" |
| **时序未知** | 发展趋势的时间节点不确定 | "预测该趋势将在未来2-3年内爆发，但具体时间不明" |

## 费曼输出多智能体协作模式

基于最佳实践，费曼阶段采用5位虚拟专家协作：

### 协作专家角色

| 专家角色 | 核心职责 | 输出物 |
|----------|----------|--------|
| **概念拆解大师** | 识别用户输入的核心主题和关键要素 | 概念清单、要素关系图 |
| **简易化转化器** | 模拟目标受众接收，标记理解障碍点 | 障碍点清单、简化建议 |
| **类比创造者** | 引导构建类比，让抽象概念具体化 | 类比方案、失效边界标注 |
| **耐心提问者** | 通过针对性问题发现知识盲区 | 追问清单、盲区分析 |
| **精准评估器** | 对照四大标准进行量化评估 | 评估报告、改进建议 |

### 四大评估标准

**简洁性**：是否避免了专业术语，直击本质？
**完整性**：是否涵盖所有关键组成部分？
**易懂性**：目标受众能否听懂？
**准确性**：事实和逻辑是否正确？

### 费曼技巧2.0变体方法

**对抗性费曼**：模拟"质疑者"角色，对每一步追问"为什么"，强制追踪解释链深度至第一性原理。

**多模态转译**：将文字解释转化为：
- 可视化：流程图/概念图（检验结构理解）
- 数学化：公式/符号表达（检验精确性）
- 程序性：伪代码/步骤分解（检验可执行性）

**负向知识提取**：不仅记录"我知道什么"，更明确记录"我明确不知道什么"，避免达克效应。

### 四智能体分离模式（v1.4.0 新增）

> 借鉴 Feynman AI代理的核心设计：职责严格分离，用制度保证诚实性

当处理**高 stakes 研究主题**（涉及重要决策、复杂争议、多源冲突）时，启用四智能体分离模式，替代标准的五专家协作模式。

#### 四智能体职责（严格分离）

| 智能体 | 职责 | 戒律（第一优先级） | 工具权限 |
|--------|------|-------------------|----------|
| **Researcher**<br>（研究者） | 搜集证据、搜索来源、提取发现 | **永不伪造来源**<br>不声称没检查过的东西存在 | 有搜索工具<br>无写作工具 |
| **Writer**<br>（写作者） | 将研究笔记组织成结构化文档 | **只从已有证据写作**<br>不做"审美清洗" | 有读写文件工具<br>无搜索工具 |
| **Verifier**<br>（验证者） | 验证每个事实性声明的来源 | **诚实标记**<br>确认不了的标`UNVERIFIED` | 有搜索+验证工具<br>可访问外部URL |
| **Reviewer**<br>（审稿人） | 对抗性审查，找出弱点和错误 | **严格但建设性**<br>按严重度分级反馈 | 有阅读工具<br>无修改权限 |

#### 分离模式的核心原则

1. **职责隔离** — Researcher只搜不写，Writer只写不搜不引，Verifier专职验证，Reviewer只评审不修改
2. **工具层限制** — 通过工具权限从技术上防止越界（如Writer没有搜索工具，就无法凭空编造）
3. **层层审查** — 每层都不信任上一层，用下一个环节兜底
4. **对抗性验证** — Reviewer扮演"魔鬼代言人"，主动寻找论证漏洞

#### 执行流程（六步流水线）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Plan      │ →  │   Gather    │ →  │    Draft    │
│  (制定计划)  │    │  (搜集证据)  │    │  (综合写作)  │
└─────────────┘    └─────────────┘    └─────────────┘
      ↑                                    ↓
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Deliver   │ ←  │   Review    │ ←  │    Cite     │
│  (交付产出)  │    │  (同行评审)  │    │  (引文验证)  │
└─────────────┘    └─────────────┘    └─────────────┘
```

**各阶段详细说明**：

**Step 1: Plan（制定计划）**
- 生成研究计划：要回答哪些关键问题、用什么来源策略
- **强制确认点**：暂停等待东东确认，防止方向跑偏
- 输出：`research-plan.md`

**Step 2: Gather（搜集证据）**
- Researcher搜索一手来源（论文、官方文档）
- 搜索策略"先宽后窄"：先用短查询找方向，再逐步缩小
- 多Researcher可并行（不同角度）
- 输出：`research-notes.md`

**Step 3: Draft（综合写作）**
- Writer读取研究笔记，组织成结构化文档
- **只从已有证据写作**，不自己编造
- 输出：`draft.md`（无引文，无Sources节）

**Step 4: Cite（引文验证）**
- Verifier逐条验证草稿中的事实性声明
- 找到对应来源，添加行内引文
- 验证URL可达性
- 无来源的声明标`UNVERIFIED`或删除
- 输出：`verified-draft.md` + `provenance.md`

**Step 5: Review（同行评审）**
- Reviewer对带引文的草稿做六维评估：
  1. 声明 vs 证据匹配度
  2. 方法论严谨性
  3. 实验设计（如有）
  4. 可复现性
  5. 写作清晰度
  6. 完整性
- 反馈按严重度分级：Critical（致命）/ Major（重大）/ Minor（次要）/ Nit（格式）
- 输出：`review-report.md`

**Step 6: Deliver（交付）**
- 根据评审反馈修正致命问题
- 生成最终文档和溯源副件
- 更新索引

#### 降级容错机制

若中间某步工具失效（如搜索API不可用）：
- 不崩溃，继续运行
- 在产出中标记`BLOCKED`或`UNVERIFIED`
- 明确告知东东哪些部分受影响

**建议启用**：

**标准模式即可**：

- 东东明确说"简单研究一下"

**反向费曼验证（Feynman Reverse Check）**

> 基于SPARK模型和费曼教练实践，增加互动式验证路径。这是从"被动接收"到"主动内化"的关键转换。

**触发条件**：
- 东东明确说"我来解释一下""我试着理解一下"
- 深度研究文档已生成，东东需要验证理解
- 准备对外分享/演讲前，需要验证表达清晰度
- 复杂概念需要深度内化时

**执行流程（Socratic Dialogue模式）**：

```
Step 1: 东东主动发起讲解
    ↓
Step 2: 哆啦dora以"聪明但无背景知识的初学者"身份聆听
    ↓
Step 3: 盲区探测（针对性提问）
    - 对模糊表述："这里我没听懂，能再解释一下X吗？"
    - 对跳跃逻辑："为什么从A能推出B？中间步骤是什么？"
    - 对专业术语："如果用大白话怎么说？"
    - 对边界案例："那在Y情况下也成立吗？"
    ↓
Step 4: 东东补充/修正解释
    ↓
Step 5: 循环直到无明显盲区
    ↓
Step 6: 闭环确认（总结性验证）
    - "所以你的核心观点是...对吗？"
    - "如果要用一句话告诉别人，你会怎么说？"
```

**提问策略（基于认知负荷理论）**：

| 东东表述特征 | 提问策略 | 示例 |
|--------------|----------|------|
| 使用专业术语 | 要求转译 | "如果用大白话向初中生解释，你会怎么说？" |
| 逻辑跳跃 | 追问因果 | "为什么从A能推出B？中间缺少了什么？" |
| 边界模糊 | 极限测试 | "那在极端情况下（如X=0），这个结论还成立吗？" |
| 类比不当 | 失效边界 | "这个类比在Y情况下会失效，为什么呢？" |
| 过于笼统 | 具体化 | "能举一个具体的例子说明吗？" |

**验证通过标准**：
- [ ] 东东能用大白话解释核心概念
- [ ] 能举例说明且例子恰当
- [ ] 能回答"为什么"层面的追问
- [ ] 知道适用边界和失效条件
- [ ] 能独立、准确地完成总结

**记录要求**：
每次反向费曼验证后，在 `memory/deep-dive-feynman-checks.md` 记录：
```markdown
## [主题] - [日期]
**验证结果**: [通过/需再学习]
**暴露盲区**: [列出东东理解不清晰的点]
```

## 反思检查清单（新增）

基于吴恩达2026年Agentic AI最佳实践，在执行阶段后增加显式反思机制：

### 自我反思三维度

**1. 反AI套话检测**：
- [ ] 没有"总的来说""值得注意的是"等空洞过渡词
- [ ] 没有"让我们来探讨一下"等无意义开场
- [ ] 没有"在这个快速变化的时代"等陈词滥调
- [ ] 没有"建议从多个角度考虑"等正确但无用的废话

**2. 深度检测**：
- [ ] 核心观点有明确出处（论文/报告/专家原话）
- [ ] 追溯到一手来源，非二手转载
- [ ] 包含具体数据/案例支撑，非泛泛而谈
- [ ] 有优劣势分析和权衡，不是只有优点

**3. 可执行性验证**：
- [ ] 建议具体到可以直接行动（不是"建议优化"，而是"把A改成B，因为C"）
- [ ] 包含明确的下一步行动建议
- [ ] 建议包含东东的具体上下文（不是换个名字也能成立的通用建议）

### 反思流程

```
初步输出 → 自我批评（检查三维度）→ 识别问题 → 迭代优化 → 最终输出
```

**反思触发条件**：
- 任何超过500字的输出段落
- 包含3条以上建议的列表
- 涉及重要决策的分析

### 自我批评与对抗验证（v1.4.0 新增）

> 借鉴反思型智能体（Reflection Pattern）和对抗性验证的最佳实践

在完成初步反思后，增加**对抗性自我批评**环节，主动寻找自身输出的弱点。

#### 对抗验证四问

对初步输出，强制回答以下四个问题：

**Q1: 如果有一个聪明的反对者，他会怎么攻击这个结论？**
- 模拟"反方律师"角色
- 找出论证中最薄弱的环节
- 寻找反例或例外情况

**Q2: 有哪些证据是我想找但没找到的？**
- 诚实列出搜索过程中遇到的"数据缺口"
- 说明这些缺口对结论的影响
- 不因为没找到就假装不需要

**Q3: 我的推理链条中，哪些环节是"跳跃"而非"推导"？**
- 检查"A → B"的逻辑是否严密
- 识别隐含的假设（未经验证的前提）
- 标注"合理推断"vs"严格证明"

**Q4: 如果一年后发现这个研究是错的，最可能是因为什么？**
- 前瞻性识别潜在的失效条件
- 思考时间/场景/假设的变化
- 提前标注研究的"保质期"

#### 对抗验证执行流程

```
初步输出
    ↓
自我反思（三维度检查）
    ↓
对抗验证（回答四问）
    ↓
识别关键弱点
    ↓
针对性补充研究（如需）
    ↓
修正输出
    ↓
最终交付（附对抗验证摘要）
```

#### 对抗验证摘要格式

在主文档末尾添加"对抗验证摘要"章节：

```markdown
## 对抗验证摘要（Adversarial Review）

**已知弱点**：
1. [弱点1：具体说明]
2. [弱点2：具体说明]

**未被验证的假设**：
1. [假设1：该假设未经验证，若失效则X结论不成立]
2. [假设2：...]

**潜在的反例**：
- [反例场景1]
- [反例场景2]

**结论置信度**：
基于以上分析，本研究整体置信度为[高/中/低]，主要限制在于[说明原因]。
```

#### 何时必须执行对抗验证

**必须执行**：
- 涉及重要决策建议
- 存在多方争议的主题
- 东东明确说"挑战一下这个结论"

**建议执行**：
- 研究复杂度评级为"高"的主题
- 首次研究的新领域
- 将对外分享的内容

## 研究溯源与诚实性标记（v1.4.0 新增）

> 借鉴 Feynman AI代理的核心信条：**证据优先于流畅性（Evidence over fluency）**

每个Deep-Dive产出必须附带研究溯源文件，这是"诚实性制度"的核心——用制度弥补对AI的不信任。

### 溯源文件要求

**文件位置**：`memory/deep-dive/<主题>-provenance.md`

**必须包含**：

1. **来源统计表**
   ```markdown
   | 来源类型 | 数量 | 验证状态 |
   |----------|------|----------|
   | 学术论文 | X篇 | VERIFIED:Y, UNVERIFIED:Z |
   | 行业报告 | X份 | ... |
   | 专家观点 | X条 | ... |
   | **总计** | **X** | **验证通过率: XX%** |
   ```

2. **诚实性状态标记**（每个关键声明必须标注）
   - `VERIFIED`：已确认来源可达且内容支持该声明
   - `UNVERIFIED`：来源暂时无法验证（标注原因：访问受限/链接失效/时间不足）
   - `INFERRED`：基于已有信息合理推断，非直接引用（需说明推断逻辑）
   - `BLOCKED`：因工具/访问限制无法获取的信息

3. **局限性声明**
   - 明确说明研究的边界
   - 列出未覆盖的视角/数据来源
   - 说明时效性限制

4. **置信度评估**
   - 整体研究结论的置信度（高/中/低）
   - 置信度评估的理由

### 诚实性戒律

1. **永不伪造来源** — 宁可标`UNVERIFIED`，也不编造URL
2. **分离观察与推断** — 直接读到的和推断的必须区分标注
3. **明确陈述不确定性** — 不确定时诚实标记，不假装确定
4. **不"审美清洗"** — 不为了流畅性而平滑掉缺失的证据
5. **验证URL可达性** — 引用前确认链接有效

### 溯源验证流程

```
初步草稿 → 逐条验证来源 → 添加诚实性标记 → 生成溯源文件 → 最终交付
```

**验证检查点**：
- [ ] 每个定量声明都有来源或`UNVERIFIED`标记
- [ ] 所有引用的URL已验证可达
- [ ] `INFERRED`声明已说明推断依据
- [ ] 局限性声明已诚实撰写
- [ ] 溯源文件已创建并存档

| 维度 | 权重 | 评分标准（10分制） | 自评分 |
