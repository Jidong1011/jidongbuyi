# RDT 方案设计参考手册

## RDT 核心概念详解

### 1. 什么是 RDT（循环深度Transformer）

传统Transformer通过**堆叠层数**增加深度：
- 16层 = 16组不同的参数
- 参数爆炸，计算成本高

RDT通过**循环迭代**增加深度：
- 1组参数 × 16次循环 = 16层等效深度
- 参数复用，计算效率提升

### 2. 三阶段架构详解

#### Prelude（前奏）— 输入标准化

**目的**：确保输入质量，为循环迭代做准备

**关键动作**：
- 需求澄清：用户真正想要什么？
- 约束识别：时间/预算/资源限制
- 目标确认：成功的标准是什么？
- 初始状态编码：生成 h₀（初始隐藏状态）

**输出**：标准化的输入向量 e

#### Recurrent Block（循环核心）— 深度迭代

**核心机制**：
```
h_{t+1} = A·h_t + B·e + Transformer(h_t, e)
```

**各组件含义**：
- **h_t**：第t轮的状态（当前认知/进度）
- **e**：原始输入（始终重新注入，防止跑偏）
- **A·h_t**：历史经验反馈（上一轮的经验）
- **B·e**：原始目标提醒（不忘初心）
- **Transformer**：本轮的深化处理

**循环终止条件**：
1. 达到最大循环次数（如3-5轮）
2. 改进幅度低于阈值（收敛）
3. 质量达到目标标准

#### Coda（尾声）— 输出沉淀

**目的**：将迭代过程中的认知沉淀为可复用资产

**关键动作**：
- 交付物整理
- 经验总结
- 方法论提炼
- 下一轮Prelude的输入

### 3. RDT vs 传统模式对比

| 维度 | 传统模式 | RDT模式 |
|------|---------|---------|
| 核心假设 | 规模决定质量 | 深度决定质量 |
| 资源策略 | 堆人/堆钱/堆时间 | 迭代深化 |
| 风险处理 | 一次性赌对 | 循环中修正 |
| 学习曲线 | 线性积累 | 阶梯跃升 |
| 适用场景 | 确定性高、标准化 | 不确定性高、需要探索 |

## 方案设计模板

### 模板A：项目规划类

#### 传统方案结构
```
1. 项目概述
   - 一句话说明
   - 项目目的（Why）
   - 项目背景（Context）

2. 阶段规划
   - Phase 1：目标、里程碑、交付物
   - Phase 2：目标、里程碑、交付物
   - Phase 3：目标、里程碑、交付物

3. 资源配置
   - 人力
   - 资金
   - 时间
   - 工具

4. 风险及应对
   - 风险1：描述、等级、应对
   - 风险2：描述、等级、应对
```

#### RDT方案结构
```
1. 核心重构点
   - 与传统方案的本质差异

2. 循环机制设计
   Prelude（输入标准化）：
   - 人才画像标准化
   - 任务需求编码
   - 初始匹配状态 h₀

   Recurrent Block（循环核心）：
   - Loop 1：目标 | 动作 | 输出
   - Loop 2：目标 | 动作 | 输出
   - Loop 3：目标 | 动作 | 输出

   Coda（输出沉淀）：
   - 交付物
   - 成长报告
   - 经验资产

3. 深度指标
   - 人均循环次数
   - 每轮改进幅度
   - 收敛度

4. 自适应机制
   - 简单场景：少循环
   - 复杂场景：多循环
```

### 模板B：流程设计类

#### 传统方案结构
```
1. 流程目标
2. 参与角色
3. 流程步骤（线性）
   Step 1 → Step 2 → Step 3 → ...
4. 异常情况处理
```

#### RDT方案结构
```
1. 流程目标
2. 参与角色
3. 核心循环（非线性）
   
   [Prelude] 输入准备
      ↓
   [Loop] 核心处理
      - 第1轮：快速验证
      - 第2轮：深化执行
      - 第3轮：优化交付
      ↓
   [Coda] 沉淀输出
      ↓
   （可选：进入下一轮循环）

4. 自适应分流
   - 简单路径：Loop 1后直接Coda
   - 标准路径：Loop 1→2→Coda
   - 复杂路径：Loop 1→2→3→Coda
```

### 模板C：策略制定类

#### 传统方案结构
```
1. 现状分析
2. 目标设定
3. 策略路径（多条并行）
   - 策略A
   - 策略B
   - 策略C
4. 资源配置
5. 风险评估
```

#### RDT方案结构
```
1. 核心假设识别
   - 列出3-5个关键假设

2. 假设验证循环
   
   对每个假设：
   Loop 1：快速验证（低成本测试）
   Loop 2：深化验证（扩大样本）
   Loop 3：确认/否定（决策）

3. 策略收敛
   - 基于验证结果，收敛到1-2条主路径

4. 执行循环
   - 小步快跑，快速迭代
```

## 常见误区与避免

### 误区1：循环次数越多越好
**错误**：认为循环越多质量越高
**正确**：3-5轮通常足够，过多会导致"过思考"
**判断标准**：改进幅度递减且趋于稳定

### 误区2：循环就是重复
**错误**：每轮做同样的事情
**正确**：每轮有明确不同的目标，层层递进
**示例**：
- Loop 1：理解问题
- Loop 2：设计方案
- Loop 3：验证方案

### 误区3：忘记重新注入原始目标
**错误**：迭代过程中逐渐偏离初衷
**正确**：每轮都回顾原始目标 e
**机制**：在SKILL.md中强制要求每轮开始时回顾目标

### 误区4：所有方案都用RDT
**错误**：生搬硬套RDT思维
**正确**：简单明确、确定性高的任务，传统方案更直接
**判断**：是否需要探索？是否需要试错？

## RDT适用性评估清单

| 问题 | 是 → 适合RDT | 否 → 适合传统 |
|------|-------------|--------------|
| 是否需要试错探索？ | ✓ | 传统直接执行 |
| 是否有不确定性？ | ✓ | 传统按流程走 |
| 是否追求深度而非广度？ | ✓ | 传统覆盖全面 |
| 是否可分解为迭代步骤？ | ✓ | 传统一步到位 |
| 资源是否受限（人/钱/时间）？ | ✓ | 传统堆资源 |
